# Small Change Delivery

以一个紧凑闭环交付中小改动。保持单 writer，不创建计划地图或多阶段代理流水线。

## 1. 确认适用范围

读取当前路径下的 `AGENTS.md`、项目规则、需求或 OpenSpec，并检查工作区状态。

仅在以下条件同时成立时继续：

- 需求、验收标准和不做内容足够明确。
- 改动能在一个主会话内完成。
- 一个 writer 可以安全修改和集成。
- 不涉及资金、核心权限、安全边界、数据迁移、复杂并发或难回滚副作用。

若存在会改变业务规则、数据、权限、API 契约或范围的关键歧义，暂停受影响部分并请求确认。若任务跨多个阶段、模块或高风险边界，转用 `$subagent-delivery`；若目的地仍不清晰，转用 OpenSpec、`$grill-with-docs` 或 `$wayfinder`。

完成标准：明确记录以下内容，并且没有阻断实现的关键歧义。

```text
范围：
不做：
验收标准：
风险：
验证：
发布权限：默认不 commit、不 push、不部署
```

## 2. 检查影响面

从入口到结果追踪真实调用链，只检查适用层：

- 入口、路由、Controller 或 UI 事件。
- Service、领域规则、状态变化和异常路径。
- Mapper、SQL、缓存、事务和外部依赖。
- DTO、VO、序列化、API 文档及前端字段绑定。
- 配置、迁移、部署兼容性和现有测试。

对每个改变的行为，确认调用方、数据来源、失败路径、兼容性和验证方式。字段存在但值异常时，区分未赋值、源数据为空、映射丢失、前端覆盖和部署未生效。

完成标准：每个计划修改的行为都有来源、消费者、异常路径和验证方式；无关代码保持不动。

## 3. 实施最小改动

- 只修改实现验收标准所需的文件和行为。
- 复用项目现有模式，不新增无必要的公共抽象、共享 helper 或配置。
- 保持无关 API contract、状态机、权限和兼容行为不变。
- 关键注释只解释非显然的“为什么”。
- 保留用户已有改动；脏工作区只处理本任务路径。

实现过程中运行最小反馈命令，尽早发现编译、类型或测试错误。

完成标准：验收范围内的生产实现完整，diff 中每一处修改都能对应需求。

## 4. 按风险验证

先确定测试缝，再选择最小但充分的验证：

- Bug 修复：优先添加能复现问题的回归测试，再修复。
- 业务规则：覆盖正常、边界、非法状态和依赖失败。
- API：覆盖参数校验、HTTP 状态、JSON contract、鉴权和异常映射。
- SQL 或持久化：覆盖条件、空结果、影响行数、事务和隔离；必要时检查执行计划。
- 前端：覆盖主要交互、异步状态、错误态和关键请求参数。
- 配置或文案：运行构建或静态检查，并记录可重复的手动验证。

Java/Spring/MyBatis 任务优先使用 `$asana-openspec-java-workflow:java-test-strategy`；复杂 Bug 可使用 `$diagnosing-bugs`；浏览器行为可使用 `$webapp-testing`。缺少环境、凭据或 fixture 时标记为 `BLOCKED` 或未验证，不把替代验证写成真实验收。

完成标准：所有适用的定向验证已通过，失败或未运行项有明确原因和影响。

## 5. 简化并复核

检查最终 diff，删除本次新增的重复、死代码、无效防御和过度抽象；可用时使用 `$code-simplifier`。简化后重跑受影响验证。

随后围绕固定 diff 进行 Review：

- 是否完整满足验收标准，且没有扩大范围。
- 是否遗漏调用方、异常路径、空值、兼容性或数据边界。
- 是否存在并发、事务、权限、安全、SQL 性能或资源泄漏问题。
- 测试是否覆盖真实行为，而不是实现细节。
- 日志、错误信息和运维验证是否足以定位失败。

低风险小改动可在当前会话复核；中等风险或涉及 API、SQL、事务、权限时，使用只读 Reviewer 或 `$code-review`。修复所有成立且属于当前范围的 P0-P3 问题，重跑相关验证，并针对新的 fixed point 重新 Review。重复此循环，直到没有已知且可执行的 P0-P3，或达到用户时限。

完成标准：最终 fixed point 有有效 Review 结论，所有已修复问题已复验。

## 6. 汇报状态

最终只汇报：

```text
完成：改动范围和关键文件
验证：命令、测试数量和结果
Review：fixed point、问题及修复、剩余 P0-P3
未验证：环境、HTTP、数据库或第三方验证缺口
风险：仍需关注的边界
Git：commit / push / deploy 状态
```

只有代码、适用验证和最终 Review 都成立时才声明完成。commit、push、rebase、部署和归档必须获得用户明确授权；本 Skill 默认不执行这些操作。
